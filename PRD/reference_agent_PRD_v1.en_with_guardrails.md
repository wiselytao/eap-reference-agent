# PRD v1.0: Reference Agent (Broker) — Hybrid RAG+

> **Version**: v1.0 (Final)  
> **Positioning**: Reference Agent (deterministic, strategy-limited broker), NOT a general Agent platform  
> **Core Goal**: Controlled, predictable, auditable orchestration of multi-source RAG queries using a finite strategy set, optionally exposed via MCP.  
> **Key Principle**: No Web Search/Crawling, no ReAct or free planning, no multi-agent collaboration, no arbitrary workflows.

---

## 0. Final Decisions (v1.0)

1. **TOOLS.md has no hot reload**: loaded at startup; changes require restart.
2. **Fork-Join allows PARTIAL delivery**: failed branches must be disclosed.
3. **External (E) is Fork-Join only in v1**: no E→V→S or E→G→V.
4. **Minimum Evidence Threshold**: SUCCESS requires **1–3 evidence items** with **mandatory locator**.
5. **Finite Strategy Set**: v1 ships with **6–8 fixed strategies**, controlled by profile allowlist.
6. **HYBRIDCOT disabled by default**: can only be enabled explicitly per profile.
7. **When E is involved, final answer is generated by Reference Agent** (using its own LLM).
8. **Router must not dynamically re-plan** based on result quality.
9. **HYBRID / HYBRIDCOT must satisfy Evidence Contract** to be considered SUCCESS; v1 does not auto-rescue.

## 1. Product Positioning & Boundaries

### 1.1 One-liner
Reference Agent is a **finite-strategy query broker** that allows enterprises to declaratively connect multiple RAG sources, while strictly limiting execution to auditable, pre-defined strategies.

### 1.2 Goals
- Enable **configurable but controlled** access to RAG sources via Profiles + TOOLS.md
- Provide **deterministic orchestration** without entering the agentic planning space
- Guarantee **evidence-backed, traceable answers**
- Support **MCP Server Adapter** so Reference Agent can be consumed as a tool
- Optimize cost/latency by preferring **Hybrid RAG built-in pipelines** for V+G

### 1.3 Non-goals
- Web search or crawling
- ReAct, self-reflection, or free planning
- Dynamic tool discovery or tool marketplaces
- Arbitrary workflow editors or custom step chaining
- Knowledge ingestion/indexing/schema management

## 2. Two-Layer Architecture + MCP Adapter

### 2.1 Layer A: Reference Agent Core Service
- Query Router (strategy selection only)
- Strategy Executor (fixed execution semantics)
- Answer Composer (mandatory for External Fork-Join)
- Audit & Trace (full replayability)
- Validation (minimum verifiable locator)

### 2.2 Layer B: Profiles (Configuration Layer)
- Enabled tools (subset of TOOLS.md)
- Allowed strategies (strategy allowlist)
- Limits & fallback policies
- HYBRIDCOT enablement and intent allowlist

### 2.3 MCP Server Adapter (Thin Wrapper)
- Exposes Core APIs as MCP tools:
  - `reference.ask`
  - `reference.trace`
  - `reference.validate`
  - `reference.capabilities`
- Contains **no routing or execution logic**

## 3. Configuration Artifacts (v1)

### 3.1 Configuration Layers (YAML or TOML)
Recommended files:
1. **System config**: `config.yaml`
2. **Tool manifest**: `TOOLS.md`
3. **Profiles**: `profiles/<profile_id>.yaml`

Priority order: Profile > config; TOOLS.md defines capabilities only.

### 3.2 config.yaml (Minimum)
- `llm`: provider/base_url/model/api_key_ref/temperature/max_tokens  
  (used for Router classification and E answer composition only)
- `runtime`: streaming_default=false, timeouts, concurrency
- `audit`: trace storage backend and retention
- `security`: secret backend, allowed profiles
- `observability`: metrics/logging

## 4. TOOLS.md v1 (Manifest: Pipeline-as-Tool)

### 4.1 Principles
- Each tool entry represents a **directly callable query pipeline**
- A Hybrid RAG project may expose multiple pipelines:
  - `VECTOR:` / `GRAPH:` / `HYBRID:` / `HYBRIDCOT:`
- TOOLS.md is read-only at runtime and versioned

### 4.2 Tool Entry (Minimum Fields)
- `tool_id` (unique, e.g. `<project>.<pipeline>`)
- `type`: `hybridrag_pipeline` | `external_mcp`
- `project_id`
- `adapter`: `hybridrag_chat_api_v1` | `mcp`
- `base_url` (for hybridrag_pipeline)
- `auth_ref`
- `pipeline_prefix`
- `capabilities`
- `constraints` (default limits)
- `evidence_contract`: `REQUIRED | OPTIONAL | NONE`
- `evidence_locator_policy`: `chat_message_ref`

## 7. Strategy Catalog v1 (Finite)

### 7.1 Principles
- Prefer **HYBRID/HYBRIDCOT** for V+G
- Any involvement of **External (E)** requires Fork-Join

### 7.2 v1 Strategy Instances
**Single-step**
1. `STR_V` → VECTOR
2. `STR_G` → GRAPH
3. `STR_H` → HYBRID
4. `STR_HCOT` → HYBRIDCOT (disabled by default)

**Fork-Join (parallel)**
5. `STR_E_FJ_H` → E || HYBRID

**Fallback**
6. `STR_FALLBACK_V`

## 8. Query Router v1

### 8.1 Inputs / Outputs
- Input: user query, profile, TOOLS.md, tool health
- Output: `strategy_id + params + rationale_codes`

### 8.2 Binding Readiness
- `required_bindings`
- `provided_bindings`
- `missing_bindings`
- `dependency_required`
- `resolution_policy`

### 8.3 Deterministic Selection Rules
1. Filter by profile allowlist
2. Filter by enabled & healthy tools
3. If External needed → `STR_E_FJ_H`
4. Else:
   - Document-only → `STR_V`
   - Relation-only → `STR_G`
   - Otherwise → `STR_H`
   - Use `STR_HCOT` only if profile allows

## 9. Quality Gate & Evidence Contract

- Router must not re-plan dynamically
- Tools with `evidence_contract=REQUIRED` must produce ≥ evidence_min
- If not satisfied → status is EMPTY or PARTIAL
- v1 default: **no automatic rescue**

---

## 10. Strategy Executor Semantics

- **SUCCESS**: evidence ≥ minimum, locator valid
- **PARTIAL**: Fork-Join partial success, disclose failures
- **EMPTY**: valid execution, insufficient evidence
- **FAILED**: mandatory step failure

Fork-Join merge rules:
- Deduplication
- Ordering by priority/confidence
- Conflict disclosure
- Evidence cap enforcement

## 17. Design Guardrails (v1 Mandatory Boundaries)

This section defines hard boundaries to prevent the Reference Agent from drifting into a general Agent or Workflow platform.

### G1. Router Must Not Perform Dynamic Re-planning
- Router selects a strategy only before execution
- Router must not re-select or adjust strategies based on execution results
- No query–evaluate–requery loops are allowed

### G2. Strategies Are Not Chainable
- A strategy instance is the smallest non-composable execution unit
- Reference Agent does not support strategy chaining or workflow definitions
- Multi-step behavior exists only within a single strategy instance

### G3. Evidence Must Not Trigger New Queries
- Insufficient evidence only affects response status and templates
- v1 does not support automatic rescue, hidden retries, or quality-driven re-queries

### G4. TOOLS.md Is Not a Tool Registry
- TOOLS.md is a static capability manifest
- No dynamic discovery, recommendation, or autonomous tool selection
- Tool usage is strictly governed by Profile allowlists and Router rules

### G5. No Platform-Level Multi-tenancy
- v1 defaults to per-tenant instance deployment
- No RBAC, quota, billing, or SaaS control plane features

### G6. LLM Must Not Drive Strategy Selection
- LLM is limited to intent classification, constrained extraction, and External answer composition
- All strategy selections must be explainable via rules and rationale codes
